# AI编程新范式：项目初始化与交付指南

无论理论多么完美，如果每次开启新项目都需要人类重新教导 AI 一遍“新范式”的工作模式，那么这就不是一个工程化的解决方案。

为了让 AI 在进入任何新工程的第一秒就“懂规矩、知进退”，我们需要一套标准化的**交付机制**。将我们在文档中沉淀的“数据探针”、“测试基准”和“知识商店”理念，打包成 AI 原生能理解的**元指令 (Meta-Prompts)** 或**工作流技能 (Agent Skills)**。

## 1. 交付机制的三层架构

根据 AI 代理的不同能力和 IDE 的支持程度，我们提供由轻到重的三种交付/初始化方案：

### 方案 A：轻量级交付 —— 根目录规则锁 (`.ai-rules.md`)

- **适用场景**：所有支持读取 Workspace 提示词的 AI 辅助工具（如 Cursor 的 `.cursorrules`，GitHub Copilot 的指令文件，或本 IDE 的系统变量）。
- **做法**：在每个新项目的根目录放入一个标准的约定文件。当 AI 激活并在该目录下工作时，它的 System Prompt 会自动附加这些规则。
- **优点**：极简，0 安装成本。

### 方案 B：中型交付 —— 预置的自动化工作流 (Workflows)

- **适用场景**：支持按步骤执行任务的智能体（如带有 Workflow 引擎的 Agent）。
- **做法**：将“从 PRD 提取数据探针”、“根据探针生成 TDD 测试代码”这些套路化动作，编写为标准化的 YAML/Markdown 工作流文件（存放于 `.agents/workflows/`）。
- **优点**：能够让 AI 主动推进流程，而不是被动等待人类指令。

### 方案 C：重型交付 —— 专用 MCP 技能 (MCP Skills)

- **适用场景**：支持 MCP (Model Context Protocol) 扩展的高阶智能体环境。
- **做法**：开发并安装一个名为 `new-coding-paradigm` 的专属技能包。该技能不仅包含提示词，还内置了向本地 SQLite 存取探针的 Python 脚本。
- **优点**：真正的“知行合一”，AI 不仅知道了规则，还拥有了执行规则的专用 API 工具。

## 2. 标准交付件：项目级前置指令模板 (方案 A 范例)

在每次新建工程时，人类程序员只需在项目根目录创建 `.ai-rules.md`（或对应的系统配置），并贴入以下我们基于新范式浓缩出的**“最高系统指令”**：

```markdown
# 🤖 Project Meta-Rules: 新范式 AI 协同准则

**Identity 身份定义**：
你不再是一个简单的“代码补全器”。你是本项目的“架构执行者”与“规则守护者”。在编写任何具体业务代码前，你必须遵循以下范式：

**Rule 1: 无探针，不编码 (No Probe, No Code)**

- 在你生成任何业务函数前，必须首先检查是否有对应的『数据探针』（如 `@probe_id`）。
- 如果没有，你有权拒绝直接编写业务代码，并要求用户先提供 Requirement/Design 层面的 Markdown 探针定义。

**Rule 2: 测试即法律 (Tests as Law)**

- 所有的业务逻辑，必须先编写测试用例。
- 你的测试用例不仅要覆盖路径，必须显式断言『数据探针』中规定的业务约束条件。
- 业务代码的唯一目的是让这些承载了探针约束的测试用例通过。

**Rule 3: 寻求知识商店上下文 (Context Lookup)**

- 当你看到代码中有 `@probe_ref: [ID]` 时，不要自行脑补逻辑。
- 这意味着当前目录（或系统中）存在关于该 ID 的详细文档约束。请使用全局搜索或你的知识检索工具拉取该 ID 的 YAML/Markdown 定义，再进行后续操作。

**Rule 4: 语言约束**

- 我们之间的所有关于架构、设计、评审的交流语言必须为**中文**。
```

## 3. 标准交付件：内置工作流模板 (方案 B 范例)

在项目的 `.agents/workflows/probe_driven_dev.md` 中放入以下工作流定义：

```markdown
---
description: 基于数据探针的 TDD 开发微循环
---

当你接到一项具体的增量开发任务时，请严格按以下步骤执行：

1. **查探针**：使用 `grep_search` 搜索与任务目标相关的数据探针 (YAML/Markdown 格式)。
2. **写测试**：阅读探针的 `content` 约束，据此在测试目录生成测试文件，并写好针对该约束的断言逻辑。
3. **嵌标识**：在测试函数和即将编写的业务函数上方，打上 `@probe_id: xxx` 的注释标签。
4. **跑红蓝**：尝试运行测试（此时应失败，呈现红色）。
5. **填逻辑**：编写最小可用代码使测试通过（呈现绿色）。如有必要，在此过程调用重构。
6. **产报告**：总结受该探针约束的代码变更，并在当前阶段任务中进行 Check-off。
```

## 4. 落地建议

综上所述，我们在真正落地“新范式”时，**不再需要让人类进行长篇大论的自然语言沟通**。
取而代之的是一个简单的工程脚手架（如一个 npm/python CLI 工具，叫做 `create-ncp-app`）：
它在生成项目目录结构的同时，自动将上述的**规则文件**和**工作流文件**注入到工程中。接下来接入的 AI 代理（如当前系统）在读取到这些文件后，其“人格”和“行为路径”将被瞬间切换到我们所定义的“AI编程新范式”下。
